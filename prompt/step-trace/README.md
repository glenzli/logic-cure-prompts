# Step Trace（步骤溯源 / 防逻辑跳跃）

简体中文

这个目录下的 Prompt 旨在防止大模型在多步推理过程中**悄悄跳过关键的中间步骤**，或将未经声明的隐含假设伪装成已知前提混入推导链。

## 📖 产生背景与作用

大模型在进行多步推理时，有一种极其隐蔽的缺陷：它经常给出一个"看起来显然正确"的结论，但如果你仔细拆解推理链，会发现中间有一步或几步被默默跳过了。模型用"显然""不难看出""自然地"这样的修辞巧妙地掩盖了逻辑断层，让结论看起来是从前提直接流淌出来的——但实际上中间缺了一座桥。

这种缺陷的机制根源在于：**LLM 的推理本质上是模式匹配而非形式化演绎。** 当训练数据中大量数学教材和论文使用"显然"来跳过"在该上下文中确实显然"的步骤时，模型学会了这种修辞模式，但它无法判断在当前的推导中，被跳过的步骤是否真的"显然"。更危险的情况是，模型会在推理链中悄悄引入一个未经声明的假设，然后基于这个假设继续推导——最终结论看起来完美自洽，但根基是悬空的。

这与我们在 `time-awareness` 实验日志中关于自回归路径确定性的思考一脉相承：**在任何自回归链条中，每一步的确定性都直接影响最终输出的可靠性。** 如果中间环节存在隐藏的逻辑断层或未声明的假设，错误会沿着推理链被放大和传播，最终以一种看起来正确但本质错误的形式呈现给用户。

**典型场景：**

- 数学证明中，模型写"由 A 显然可得 B"，但实际上从 A 到 B 需要一个非平凡的引理。
- 系统设计推演中，模型在"如果 X 满足条件 Y"的假设下进行了长链推导，但从未声明条件 Y 是否成立。
- 故障排查中，模型将"可能是原因 A"在下一步直接当作"原因就是 A"继续推导。

该 Prompt 通过**逐步显式化 + 假设暴露 + 断点自检**三重机制，强制模型将每一步推理都暴露在阳光下，消除隐藏的逻辑暗箱。

## 🌟 核心功能

- **逐步显式化**：禁止使用"显然""不难看出"等跳跃性表述，每一步推理必须有明确的推导依据。
- **假设暴露**：任何未经证明或未由上下文直接给出的前提，必须显式标注为`[假设]`，不允许伪装成已知事实。
- **断点自检**：推理完成后回溯检查步骤之间是否存在逻辑断层，发现断层时必须显式标注并补充。

## 💡 适用场景

- **数学证明与公式推导**：确保每一步都有严格的依据，不跳步。
- **系统设计的架构推演**：暴露设计决策中的隐含假设，防止基于未验证前提进行长链推导。
- **故障排查与根因分析**：防止将"可能性"悄悄升级为"确定性"。
