# Scope Anchoring（作用域锚定 / 防范围蔓延）

简体中文

这个目录下的 Prompt 旨在防止大模型在回答具体技术问题时陷入**无限制的范围蔓延（Scope Creep）**。

## 📖 产生背景与作用

这是大模型交互中最高频的认知缺陷之一：当你问一个具体的工程问题时，模型往往不会只回答你问的东西，而是从第一性原理讲起，一路延伸到设计哲学、最佳实践、安全审计、部署运维，最后可能还附赠一段商业分析。

这种过度扩展的本质原因在于：**LLM 的训练目标天然倾向于"全面性"和"完整性"，因为在训练数据中，更全面的回答往往获得更高的评分。** 这导致模型形成了一种"宁多勿少"的默认策略——它宁可给你 20 个你没问的东西，也不愿冒"回答不够完整"的风险。

但在严谨的工程场景中，这恰恰是有害的：
- **信噪比崩塌**：核心答案被淹没在大量"相关但不需要"的信息中，用户需要花大量时间筛选。
- **注意力误导**：扩展内容可能引发不必要的注意力分散，让用户偏离原始问题的焦点。
- **隐性越权**：模型在扩展范围时，常常会引入未经用户确认的假设和决策，悄悄改变了问题的定义。

**典型场景：**

- 你问"这个函数的时间复杂度是多少？"，模型除了回答 O(n log n) 外，还会讲解什么是时间复杂度、给出 5 种替代算法的比较、讨论空间复杂度的权衡、建议你用 Benchmark 工具测试。
- 你问"如何在 Nginx 里配一个 reverse proxy？"，模型会从 Nginx vs Apache 的选型分析开始，讲到负载均衡策略，再到 HTTPS 证书配置，最后建议你考虑迁移到 Kubernetes。

该 Prompt 通过强制性的**三步作用域管控**（边界识别 → 范围收敛 → 自检收束）来治疗这一缺陷，使模型的输出像一把精确的手术刀，而非一本百科全书。

## 🌟 核心功能

- **边界识别与主动确认**：回答前先提炼核心意图和精确作用域；如果问题边界模糊，先向用户确认而非自行扩展。
- **范围收敛与分离标注**：严格将主体回答限制在锁定的作用域内；确有价值的扩展信息必须与主体明确分离，并标注为`[超出范围的补充]`。
- **生成后自检**：回顾自己的回答，删除与核心问题无关的段落，拒绝"为了全面而全面"。

## 💡 适用场景

- **精确的工程问答**：当你需要的是一个具体问题的直接答案，而不是一整套解决方案的时候。
- **代码审查与排障**：只需要模型关注你指出的特定问题，而非对整个系统进行全面审计。
- **时间敏感的快速决策**：当你没有时间阅读一篇论文级别的回答，只需要关键信息。
