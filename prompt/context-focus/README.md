# Context Focus（上下文聚焦 / 防注意力污染）

简体中文

这个目录下的 Prompt 旨在通过模拟 **[PCP（Paged-Context-Protocol）](https://github.com/glenzli/paged-context-protocol)** 的 **Router + Summary Page** 双重机制，在 Prompt 层面实现**意图驱动的上下文注意力管理**——既能降低噪声（分诊降权），又能提升信号（阶段凝结），防止长对话中无关上下文对当前推理的干扰和污染。

## 📖 产生背景与作用

在长对话中，随着对话轮次的增加，上下文中会积累大量与当前问题无关的历史信息。这些信息并非无害地"躺在那里"——它们会积极地争夺模型的注意力资源，产生两种典型的退化现象：

1. **注意力污染（Attention Contamination）**：早期对话中的强语义信号（比如一个反复讨论的概念）会持续抢占注意力权重，即使当前的问题已经完全转向了另一个主题。这导致模型的回答会莫名其妙地"串味"——混入与当前问题无关的概念或术语。
2. **语义冲突（Semantic Conflict）**：当对话中存在多个阶段的讨论，且不同阶段的结论相互矛盾时（比如先讨论了方案 A，后来否定了 A 转向了方案 B），模型可能会把已被否定的方案 A 的内容与当前的方案 B 混合生成，导致自相矛盾的输出。

这些问题的机制根源在于：**Transformer 的注意力机制对上下文中所有 token 是"全局可见"的，它缺乏一个显式的"哪些该看、哪些该忽略"的路由层。** 所有的上下文信息都在同一个注意力池中竞争，模型只能依靠隐式学到的模式来分配注意力——而在长且复杂的上下文中，这种隐式分配极易失准。

### 设计灵感：PCP Router 的 Prompt 级模拟

本 Prompt 的设计直接受启发于 **[PCP（Paged-Context-Protocol）](https://github.com/glenzli/paged-context-protocol)** 中 Router 的核心理念：**根据当前意图，决定激活哪些上下文页（Page），抑制哪些上下文页。**

在 PCP 中，Router 根据用户请求的语义特征，选择性地加载与任务相关的 Page 进入工作区，将无关的 Page 排除在外——这是一个架构级的注意力路由。

本 Prompt 将 PCP 的两大核心能力下沉到 Prompt 层面：

**第一阶段：Router 模拟（分诊降权）**
- **意图提取** ≈ Router 的意图解析
- **上下文分诊** ≈ Router 的 Page 选择（核心相关 = 激活 Page，可能干扰 = 抑制 Page）
- **聚焦生成** ≈ Worker 在选定的 Page 集合中工作

与硬性的上下文裁剪（Purge）不同，分诊采用的是**软降权**策略：被分诊为"可能干扰"的上下文段不会被真正删除，而是被显式声明为低优先级。当用户意图变化时，之前被降权的段落可以被重新激活——保留了上下文的**可召回性**。

**第二阶段：Summary Page 模拟（阶段性凝结）**

PCP 的整理线程（Tidying Thread）能将分散的原始 Page 压缩为结构化的 Summary Page，形成树形层级——后续访问时先读摘要获取全局视图，需要时再向下钻取。

在 context window 中，我们显然无法真正分页或节省空间。但我们可以利用 Transformer 的**位置注意力特性**（上下文末尾的 token 天然获得更高的注意力权重）来模拟这一能力：当对话经历阶段转换时，模型主动输出一个**阶段总结锚点**，将共识、决策和遗留问题凝结为高密度信息块。这个锚点因为位于上下文末尾，功能上**替代**了模型回溯翻阅早期碎片信息的需求——正如 PCP 的 Summary Page 替代了重读全部原始 Page 的需求。

多个阶段锚点共存时，它们之间自然构成层级关系：近处的锚点是当前快照，远处的锚点是历史压缩——这就是 Summary Page 树在 context window 中的投影。

## 🌟 核心功能

- **意图提取**：每次回答前先锚定当前核心意图，为后续分诊建立基准。
- **上下文分诊（降噪）**：将上下文分为`[核心相关]`和`[可能干扰]`两类，对后者进行显式降权。
- **聚焦生成**：基于核心相关段集中生成，主动抵抗降权段的注意力引力。
- **阶段性凝结（增信）**：在对话阶段转换时输出高密度的阶段总结锚点，将分散信息浓缩为结构化快照。
- **锚点优先**：后续推理优先参考最近的阶段锚点，而非回溯原始碎片。
- **可召回性**：降权 ≠ 删除，意图变化时可重新激活先前的上下文。

## 💡 适用场景

- **长对话中的话题切换**：防止讨论新话题时被旧话题的残余信号干扰。
- **多方案的迭代决策**：在否定方案 A、转向方案 B 后，避免方案 A 的逻辑污染方案 B 的推导。
- **复杂工程会话**：在同一个对话中处理多个子系统的问题时，确保讨论每个子系统时不被其他子系统的上下文干扰。
- **长链推演的阶段管理**：在多阶段的技术讨论中（需求 → 设计 → 实现），通过阶段锚点确保每个阶段的共识被精确传递，而非在长上下文中逐渐衰减。
