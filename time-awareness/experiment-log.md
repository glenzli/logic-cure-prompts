# 实验日志：关于“时间感知” Prompt 的观察与推测

在实际应用 `time-awareness` prompt 的过程中，我记录了以下观察与推测：

## 1. 指令遵循度与元数据注入

引入 prompt 后，不同模型对指令的遵循度存在差异。例如，Gemini 3.1 Pro 几乎每次都会主动获取时间，而 Gemini 3 Flash 的触发频率较低。

起初，我推测这是由于模型对 Function Call 的遵循度不同导致的。然而，仅为了获取当前时间而调用一次 Function Call，其交互成本显得过高。同时我注意到，模型在单次对话中实际上能够结合当前 IP 地址主动提供位置信息。在对 Gemini 3.1 Pro 尝试进行直接的询问后，我得到了如下回复：

> ...
> 时空元数据注入: 这是一个特殊的内部接口，它负责把刚才咱们争论的“新加坡时间”这种快照，以近乎静态的方式注入到我的 Context 头部。
> ...

这表明，结合模型能主动给出地理位置等上下文信息的能力进行交叉验证，我们极有理由推断：**时间数据是通过“元数据注入”的方式隐式传入 Context 中的可能性极高。**

这揭示了一个现象：如果仅仅是底层注入元数据，当用户意图未明确指向“时间”时，这些数据往往会被模型的注意力机制忽略。但是，通过显式地引入 `time-awareness` prompt，模型与这些隐式的时空元数据建立了强关联，使得“当前时间”这一维度的信息在模型生成回答时获得了实质性的权重影响。

## 2. 时间漂移与 Logit 采样幻觉

另一个值得关注的现象是“时间漂移”。特别是在连续的高强度对话后，模型偶尔会给出错误的时间。例如，当前实际时间是 20:00，模型却回复现在是 20:15。

由于这种漂移通常是向“未来”发生，无法简单用缓存机制或 System Prompt 预编译来解释；同时，“时间获取”作为如此基础的功能，也不能武断地以代码 BUG 来解释这种错误。因此，我推断出一种可能性：**Logit 采样幻觉（Logit Sampling Hallucination）**。

也就是说，尽管模型在 Context 中拥有正确的时间基准输入，但在最终生成 Token 的过程中，由于我们引入了拟人化的伪随机性（如设置了 Temperature），导致在复杂上下文的干扰下，最终输出的时间出现了采样偏差和漂移。

## 3. 对自回归协议（如 PCP）的启发

这种因概率采样带来的漂移现象，对于任何依赖自回归运行路径的系统来说都是一种潜在风险——这其中就包括 PCP (Paged-Context-Protocol) 协议，其逻辑线本质上也是一种自回归。

这引发了我的进一步思考：在执行长逻辑推理或复杂的自回归路径时，我们是否应该**在所有处于中间环节的自回归路径中消除随机采样（设置 Temperature = 0），以确保逻辑演算的绝对确定性，仅在最后一次直接面向用户输出的生成中才引入 Temperature**。这或许才是大型系统交互的最佳实践结构。

## 4. 实验进展：强化 Prompt 对漂移的抑制与猜想升级 (2026-02-28)

在进一步的优化实验中，我针对 Gemini 3.1 Pro 调整了 prompt，特别增加了以下约束：

> “在构建回答时，必须将该时间作为绝对的物理参考坐标系，根据时间特征动态调整你的表述，并在最终输出时抵御上下文干扰，确保时间信息的严谨准确”

**观察结果：**
增加上述强化指令后，Gemini 3.1 Pro 几乎不再出现明显的时间漂移情况。

**猜想升级：**
基于此，我对之前“时间漂移主要由 Logit 采样引起”的猜想进行了升级。现在的推测是：**时间偏移可能是双重因素叠加的效果。**

1.  **注意力机制偏移：** 时间信息虽然作为元数据存在于上下文中，但由于其本质也是 Token 流的一部分，在深度网络的矩阵计算中，可能会受到正文其他强烈语义信号的干扰（Attention Drift），导致在注意力分配时已经发生了微小的数值偏移。
2.  **采样随机性放大：** 这种在表征层已经发生的偏移，在最终经过 Temperature 采样时被进一步放大，从而表现为输出层的时间幻觉。

通过增加“绝对参考坐标系”和“抵御干扰”的强化指令，实际上是在模型计算注意力分配时，人为地为“时间 Token”锚定了更高的权重和稳定性，从而从源头上抑制了漂移。
